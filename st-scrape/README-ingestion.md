# Swimtrends Ingestion Platform — Runbook

Deploys the `st-scrape` scraper to AWS. Registered meets are scraped
automatically ~6h after they finish; raw JSONL lands in
`s3://swimtrends-meet-data/raw/meet=<id>/`.

## Prerequisites

- AWS credentials for account `179537025528` (region `eu-west-1`).
- Docker running (CDK builds the scraper image + the Lambda bundle as assets).
- Python env for CDK: `cd swimtrends-app && python -m venv .venv &&
  .venv/bin/pip install -r requirements.txt`.

## Deploy

```bash
cd swimtrends-app
# alert_email subscribes an address to the SNS alert topic (confirm via email).
.venv/bin/cdk deploy SwimtrendsIngestionStack -c alert_email=you@example.com
```

The existing `SwimtrendsAppStack` (S3 bucket, Glue, Athena) is untouched.

## Configure the CLI

The `swimtrends` CLI reads two environment variables:

```bash
export REGISTRY_TABLE=swimtrends-meet-registry
# The dispatcher Lambda's physical name is generated by CloudFormation; read it back:
export DISPATCHER_FUNCTION=$(aws cloudformation describe-stack-resources \
  --stack-name SwimtrendsIngestionStack \
  --query "StackResources[?starts_with(LogicalResourceId, 'Dispatcher') && ResourceType=='AWS::Lambda::Function'].PhysicalResourceId | [0]" \
  --output text)
```

Run the CLI from the scraper dir: `cd st-scrape`.

## Register meets

```bash
python -m ingestion.cli register 10970 --categories DM-L,DMJ-L --end-date 2024-07-11
```

Optional per-meet overrides: `--grace-hours N` (default 6, hours after the meet's
last day at 23:59 local before scraping) and `--deadline-hours N` (default 72,
after which the meet is force-scraped even without confirmed results).

Registering a `meet_id` that already exists is refused (to avoid clobbering its
status); re-arm an existing meet instead with `--rescrape`:

```bash
python -m ingestion.cli register 10970 --rescrape   # status->scheduled, attempts=0, error cleared
```

`--rescrape` needs no `--categories`/`--end-date` (it keeps the existing values).

## Dispatch

```bash
python -m ingestion.cli dispatch                 # normal due-check cycle (all scheduled)
python -m ingestion.cli dispatch 10970 --force   # scrape one meet now, skip the gates
python -m ingestion.cli dispatch --all --force   # backfill: scrape ALL scheduled meets now
```

The hourly EventBridge schedule runs the same due-check cycle automatically;
manual dispatch is for backfill and ad-hoc runs. As a safety guard, a bare
`dispatch --force` (force, no meet, no `--all`) is refused — force-scraping the
entire registry must be spelled `dispatch --all --force`.

## Historical backfill

```bash
python -m ingestion.cli register 10329 --categories DO --end-date 2023-07-09
python -m ingestion.cli register 10969 --categories DO --end-date 2024-07-14
# ... register all historical meets (end_date in the past) ...
python -m ingestion.cli dispatch --all --force
```

Past `end_date` meets dispatch immediately with no completeness polling.

## Status lifecycle

`scheduled → scraping → scraped | failed`. A `failed` meet is retried each cycle
up to 3 attempts, then alerts and stops — re-arm with `register <id> --rescrape`.
Inspect: `aws dynamodb scan --table-name swimtrends-meet-registry`.

## Where things land

- Raw data: `s3://swimtrends-meet-data/raw/meet=<id>/{meet_info,races,results}.jsonl`
  (bucket is versioned → prior scrapes retained automatically).
- Scrape logs: CloudWatch, log stream prefix `scrape`.
- Dispatcher logs: CloudWatch, the Lambda's log group.
- Alerts: SNS email on scrape success, failure, and deadline-forced runs.

## Cost

Idle ≈ S3 storage only. Fargate bills per-second during a scrape; Lambda,
EventBridge, DynamoDB on-demand, and SNS are negligible at this volume. There is
**no NAT gateway** — the scrape task uses a public subnet + public IP.
